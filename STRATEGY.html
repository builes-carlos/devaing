Why devaing exists

The framework is a deliberate synthesis of specific ideas applied to a specific context. This section documents the sources, what was taken from each, and the selection criteria.

The core problem

Vibe coding

Feels like magic: instant software from natural language. But the result is accumulated technical debt with no structure, no institutional memory, and no way to collaborate beyond the current session. The agent doesn't know what it built yesterday.

Core problem: no context rationalization between sessions.

Heavy process (Jira + PRDs + specs)

Designed for human coordination in large teams. Produces artifacts nobody reads, duplicates information that drifts from the code, and burns more tokens on coordination than on building. Doesn't scale down to 1-3 person teams.

Core problem: overhead designed for teams of 20, applied to teams of 1.

The devaing thesis

GitHub Issues is not bureaucracy. It is a token rationalization system: it enables granular work (one session per issue), parallel collaboration (multiple agents or people taking different issues), and gives agents enough context to execute without re-explaining from scratch. The overhead that gets eliminated is human coordination. What stays is context continuity between sessions.

The abstraction layer

devaing skills are abstract classes: stable contracts with swappable implementations.

AGENTS.md / CLAUDE.md → interface: stable, never changes

devaing skills → concrete class: your code, you update it

ce-work, grill-me, etc. → dependencies: used internally, invisible outside

AGENTS.md says /devaing-work. It doesn't know what's inside. If tomorrow something better than ce-work appears, you update the skill's SKILL.md. Nobody who calls /devaing-work notices. The contract is fulfilled the same way.

This is why devaing skills must not modify third-party skills directly. You own your abstraction layer; they own their implementations. If the methodology changes, you update the skill, not AGENTS.md.

Sources and selection criteria

General criterion: take what reduces friction for 1-3 person teams working with agents. Drop what exists to coordinate humans in large organizations.

📐
Matt Pocock — structured agent development methodology for teams

Taken

  • ✓GitHub Issues as the atomic unit of work. One issue = one isolated worktree = one agent session. Eliminates shared context problems, enables real parallelism.
  • ✓Canonical triage labels. Agents filter their own work without human intervention.
  • ✓Post-implementation ADRs for non-obvious decisions. Not planned upfront — documented after, when the trade-off is actually known.
  • ✓Vertical slices: each issue describes end-to-end behavior, not layers. UI + logic + tests as one unit.
  • ✓Compound Engineering as the execution layer (ce-work, ce-frontend-design, reviewers). No reinventing the implementation wheel.

Dropped and why

  • ✗PRDs as living artifacts. In Pocock, the PRD is the team's source of truth. For a solo builder it's an intermediary that drifts from the code. Replaced by CONTEXT.md + issues with acceptance criteria.
  • ✗Extensive quiz loops before generating issues. Pocock confirms every decision with the human. For speed, devaing validates only two points: domain discovery (grill-me) and the epic list. The rest the agent decides.
  • ✗One PRD per epic. Produces 80k+ token sessions before writing a line of code. Replaced by direct issue generation from grill-me context — or progressive slice creation as work proceeds.
⚡
gstack (Garry Tan / YC) — role fragmentation for solo founders, feed-forward pipeline

Taken

  • ✓Feed-forward pipeline: each step writes artifacts the next consumes. grill-me → CONTEXT.md → epics → issues → ce-work → CONTEXT.md update. No step starts cold.
  • ✓Role chaining via CE: ce-frontend-design (designer), ce-work (engineer), ce-code-review (reviewer). The founder plays all roles through specialized skills, not manually context-switching.
  • ✓Prototype before committing to UI. Validates navigation flow before generating issues. In devaing: prototype survives as a living skeleton, not deleted after validation.
  • ✓Checkpoint/context-restore concept → adapted as mid-flight session recovery in devaing-work. If a session dies, the next invocation detects the orphaned branch and resumes.

Dropped and why

  • ✗Taste learning (GBrain): persists which design variants the user prefers with weekly decay. Interesting but CONTEXT.md + memory system already cover the relevant part for devaing's scope.
  • ✗Multi-model validation (/codex): same code sent to OpenAI for cross-vendor check. devaing's target is speed, not redundancy.
  • ✗Safety gates (/freeze, /careful): devaing-work already verifies blockers before executing, which covers the most common destructive scenario.
🧠
GSD — get-shit-done (gsd-build/TACHES) — context rot prevention, isolated subagents per task

Taken

  • ✓Context rot model: 0-30% context = peak quality, 50%+ = model rushes, 70%+ = hallucinations and dropped requirements. Every architectural decision in the build loop traces back to this.
  • ✓Progressive working mode: don't generate the full backlog before building. Issues emerge as work proceeds, keeping the init session lean. The reason is context rot, not preference.
  • ✓Context budget warning in devaing-work: before dispatching to ce-work, if the session is already loaded from prior slices, warn the user and recommend a fresh session.
  • ✓Fresh session suggestion after each closed slice: the implementation context of one slice contaminates the next. Each slice runs better in a clean 200k token window.

Dropped and why

  • ✗Autonomous next-step detection (/gsd-progress --next): devaing requires the human to pick the milestone. Deliberate — solo builders want control over what gets built next, not full autonomy.
  • ✗Separate REQUIREMENTS.md, STATE.md, ROADMAP.md: GSD splits state into four files. devaing unifies in CONTEXT.md + GitHub Issues. Less surface area, same information.
🤖
Compound Engineering — framework-agnostic execution layer for code agents

Taken

  • ✓ce-work as implementation engine: isolated worktree, TDD, PR, CI, merge. devaing wraps it with CONTEXT.md updates and ADR management.
  • ✓ce-frontend-design for UI issues: visual quality implementation verified via screenshots, iterating until expected behavior is met.
  • ✓Specialized reviewers (ce-code-review, /ultrareview) as quality gate before merge, run in fresh context to avoid confirmation bias.
  • ✓/improve-codebase-architecture every 2-3 milestones: detects drift between the emergent architecture and what was planned. Real systems never evolve exactly as designed.

Dropped and why

  • ✗Modifying third-party skills to adapt them. Version dependency becomes fragile and maintenance bifurcates. devaing invokes skills as-is; orchestration logic lives in devaing skills.
  • ✗Running reviewers in the same implementation context. CE reviewers are designed for fresh context — running them in the same worktree defeats the purpose of independent peer review.

The synthesis

From Pocock

The structure: atomic issues, isolated worktrees, ADRs, Compound Engineering

From Gary Tan / YC

The speed: bias for action, minimal human validations, real QA on real features

Original to devaing

CONTEXT.md as persistent memory between sessions. grill-me as a single domain discovery investment. Mandatory admin panel. Prototype as a living skeleton.

The cut criterion in every decision was always the same: does this reduce tokens consumed before producing working software, or does it increase them? Everything that increases them without direct product value was removed.

When NOT to use devaing

  • —Large teams with dedicated PMs and designers: Pocock is the right choice. devaing assumes the same builder who writes the issue also validates the QA. With separate roles, the coordination overhead devaing eliminates is actually necessary.
  • —Projects skipping grill-me: devaing without domain discovery is vibe coding with extra steps. The investment in grill-me is not optional — it's the foundation of everything that follows.
  • —Regulated domains requiring formal audit trails: devaing ADRs are post-hoc and lightweight. If the domain requires decision traceability before implementation (fintech, healthcare, government), a more rigorous design process is needed.

Hyper-agile framework for building products with AI

For solo builders, dev pairs, and early-stage startups who want product fast without heavy process.

Key design decisions

✗

No PRDs

PRDs are expensive intermediaries nobody reads. Issues with acceptance criteria are the spec. ADRs capture non-obvious decisions post-implementation.

✗

No quiz loops

Only 2 human validations in devaing-init: grill-me (domain) and epic list approval. Everything else the agent decides without asking.

✓

Phase lifecycle — not a flat backlog

A project has phases (MVP, Beta, V2…). /devaing-init runs once for setup, then calls /devaing-phase for Phase 1. Subsequent phases are started manually after the previous one closes. /devaing-phase is blocked while any issue in the current phase is open — enforced automatically.

✓

Two upfront questions (per phase)

Each phase asks: (1) Granularity — Broad (2-4 slices/epic), Balanced (3-6), Detailed (6-12). (2) Tracking — GitHub or Local (BACKLOG.md). Granularity can differ between phases — Broad for MVP, Detailed for a compliance-sensitive Phase 2.

✓

Prototype as a living skeleton

UI prototypes are not deleted after UX validation. They survive as navigation skeletons. Each devaing-work slice replaces one mock screen with real implementation. The rest of the prototype stays intact as scaffolding for future slices.

✓

Milestones = epics

GitHub milestones group issues by epic. The human navigates by milestone, not flat list. Direct correlation with what was seen in discovery.

✓

CONTEXT.md as single source of truth

Updated after every operation. Never stale because it grows with the product. Includes glossary, architecture, constraints, epics, and UX conventions.

✓

Mandatory admin panel

Every project must have an admin panel epic, even if the user didn't mention it. Guarantees all system configuration lives in the DB, never hardcoded.

Granularity (Step 0, Question 1)

Broad — 2-4 slices/epic

Each task covers a lot of ground. 2-3 high-level acceptance criteria. Move fast. Great for MVPs and exploring new ideas.

Balanced — 3-6 slices/epic

One task = one concrete thing a user can do. 4-6 specific acceptance criteria. The natural middle ground for known-scope projects.

Detailed — 6-12 slices/epic

Every edge case has its own task. Full acceptance criteria with negative paths. For projects where bugs have real consequences.

Tracking (Question 2)

GitHub

Issues + milestones. Visibility, CI integration, collaboration.

Local

BACKLOG.md with checkboxes. No external accounts. Start immediately.

/devaing-phase

Kick off a new phase: discovery, epics, prototype, and all issues for the phase.

/devaing-phase "Phase name" e.g. "MVP", "Beta", "Compliance V2"

Blocked while a phase is open

Step 0 checks for any open issues in the current phase. If found, lists available skills and stops. Cannot start a new phase until all issues are closed.

0

Phase gate

Count open issues across all current-phase milestones. If > 0: list available skills, stop. If 0: proceed.

1

Phase name + granularity

Confirm phase name (passed as argument or asked). Ask granularity: Broad (2-4 slices/epic), Balanced (3-6), Detailed (6-12). Granularity can differ per phase.

2

Read context

Read CONTEXT.md completely. Read all open and closed issues from previous phases. Build a full picture of what was built, what is open, what was deferred.

3

Model gate (upgrade)

Suggest switching to a more capable model before discovery. Domain discovery is the highest-value step — context quality here propagates to every subsequent issue. Wait for user confirmation.

4

Discovery

Phase 1 (grill-me): full domain interview. Everything: problem, users, business model, competitors, what it does NOT do. Invite the user to add files, images, docs, screenshots. Let grill-me run its full loop.
Phase 2+ (incremental): "Phase N-1 is done. Here's what we built. What's new? What changed since then?" Focused interview on deltas — new requirements, pivots, deferred backlog items now in scope.
5

Update CONTEXT.md

Populate or extend: Project, Domain glossary, Architecture, Key constraints. Also update ## Phases table with new phase row (status: In Progress). Commit.

6

Define epics

Derive epics from the interview. Present list for approval — one validation checkpoint. Add any missing epics based on domain knowledge (auth, admin panel, notifications, etc.).
Admin panel rule: if admin panel is not in the user's list, add it automatically. Every product with a DB needs one. Let the user remove it — don't wait to be asked.
Annotate next-phase backlog: epics identified in discovery but out of scope for this phase. Stored in CONTEXT.md ## Next phase backlog.
Create GitHub milestones for each approved epic (GitHub mode) or BACKLOG.md sections (Local mode).
7

Prototype

Phase 1: invoke /prototype for all UI epics. Creates a living navigation skeleton — never deleted.
Phase 2+: extend the existing skeleton with new screens for new epics. New screens are mocks. Previous screens (already implemented) are left intact.
Non-UI epics: skip prototype for those epics only.
8

Model gate (downgrade)

Suggest switching to a faster/cheaper model for issue generation. The creative work is done — issue generation is mechanical. Wait for user confirmation.

9

Generate issues

Create all issues for every epic, calibrated to the chosen granularity. Dependency order — infrastructure and auth slices first. Each issue: what to build, acceptance criteria, blocked by. Label: ready-for-agent or ready-for-human.

/devaing-work

Implement an issue end-to-end and keep CONTEXT.md alive.

/devaing-work #N — takes a specific issue
/devaing-work <milestone> — determines and creates the next slice, then implements it
1

Detect mid-flight session (from gstack checkpoint)

GitHub mode: look for open issue in the milestone with an existing branch. If found: offer to resume.
No-GitHub mode: look for unmerged branch matching the milestone slug. If found: offer to resume.
If no mid-flight found: determine next slice, create issue (GitHub mode) or name branch (no-GitHub mode), proceed.
2

Resolve the issue

If given #N: read the full issue, verify all blockers are closed. If given a milestone name: read closed issues and CONTEXT.md, create the next slice.

3

Context budget check (from GSD context rot model)

If this is not the first slice of the session: warn the user that running ce-work in a loaded context degrades quality (50%+ = rushes, 70%+ = hallucinates). Recommend starting a fresh session. User confirms to continue.

4

Route by type

PROTOTYPE CHECK

If a prototype exists and the issue involves UI: identify the corresponding screen. Pass it to ce-frontend-design as reference. Replace that mock with real implementation. Leave all other screens intact.

UI issue: ce-frontend-design first (with prototype screen), then ce-work for remaining logic.
Non-UI issue: ce-work directly.
5

Update CONTEXT.md and ADRs

After merge: evaluate changes in glossary, architecture, constraints. Ask about non-obvious decisions. If yes: create ADR in docs/adr/NNNN-slug.md. Also ask about known limitations left incomplete.

6

Close the issue + phase completion check

Close issue with PR reference. Check if this was the last open issue in its milestone.

Milestone closed, other milestones still open: show next milestone, suggest fresh session.
ALL phase milestones at 0 open issues — phase complete: update CONTEXT.md ## Phases (In Progress → Complete), commit. Show QA prompt. Suggest /improve-codebase-architecture every 2-3 phases. Show command for next phase: /devaing-phase "next-name".
